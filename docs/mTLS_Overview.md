# mTLS Overview

An overview of the mTLS protocol designed for use in the trailblazers application, between the server and the physical devices. Focuses on complete security from end to end, making it fundamentally impossible to spoof or sniff a connection.

Devices must be manually registered before they can send data.

A library called mbedTLS provides most of the functionality for the device networking here. It is assumed this will be used, but it is not mandatory as long as all of it's functionality can be replicated with whatever method the MSD team chooses. 

## 1.) Initial device Registration

To securely register, both the device and the server need to know a few pieces of information about the other innately.

### The server must have

- A self hosted certificate authority
  - step-ca is an open source straightforward option
  - hosted in an AWS ec2 instance
- A root certificate generated by the certificate authority(CA) that authenticates the server
  - .pem file
  - stored on the server as well

### The device must have

- A unique device ID integer value
  - This is a public value
  - Must be physically accessible on the device(barcode, label, or qr code)
- A unique serial number value
  - This is a private value, store securely
  - Must be physically accessible on the device(barcode, label, or qr code)
  - The hashed mac address of the device is the agreed upon value to use here.
    - This might not be secure enough as a mac address is public info. Look into if possible. 
- A copy of the certificate for the server from the CA
  - .pem file
  - This is private, store securely
  - This must be baked into the device from the start
- An internal clock that tracks the current unix timestamp
  - How this gets set and how regularly validated it is, is up to MSD

### Regestration Process

#### Serverside Registration from Trail Manager

- Trailblazers website frontend features a devices page
  - Only accessible to those with the requiste permissions to add devices
- Register new device option is accessible from this page
- Register new device prompts user to input device ID and serial number
- Upon submission, a new entry in the registration table is created, along with a corresponding new entry in the device table
  - Registration table fields-
    - registration_id: number(int, PK), created on new entry, primary key
    - device_id: number(int, FK), created on new entry, foreign key for the device table entry, corresponds to the device id value
    - date_registered: number(UNIX timestamp), unix timestamp of the initial registration process completion, filled in on first device connection
    - date_cert_issued: number(UNIX timestamp), unix timestamp of the latest cert issue date, filled in when the CA creates a new certificate for the device
    - cert_time_to_live: number(int, seconds), certificate time to live, expiration date is `date_cert_issue + cert_time_to_live`
- Also upon submission, the device serial number is stored in AWS secrets manager
  - Secrets manager uses JSON key value pairs
  - For this, it's device ID, device serial
  - device ID can be used to fetch the serial number as needed
  - This can NOT be seen at all on the website at any point. Once it's entered, it's sealed away and can't be edited unless the device is deleted and re-entered.

The website here is alraedy secured via traditional internet protocols, and the serial number will never be revealed, ensuring we can consistently identify that a device is legitimate. Even if a trail manager gets hacked and registers their own fake devices, 

#### Device Initial Boot

Upon first boot of the device the following occurs(or subsequent boots if the first boot wasn't successful at executing the steps below)

This is ran once and once only upon successful completion.

- Device updates its internal clock by fetching the current unix timestamp from a time server
  - No prefrences on how this is done, do whatever is standard
- mbedTLS generates a public/private key pair for the device
  - Private key stored securely
- mbedTLS uses the key pair to generate a certificate signing request(CSR)
  - Common name field should be public device ID
  - Other fields optional but most likely unused
- Device creates an object of {device ID, current unix timestamp, CSR}
- mbedTLS hashes the above object against the device serial number
  - HMAC-SHA-512 is the hashing method of choice here
- Hash is used to create the final object of {{device id, current unix timestamp, CSR}, hash}
- Device performs basic TLS handshake on registration API endpoint
  - Device's stored root certificate is used to verify connection
- Device sends GET request to specific API registration endpoint
  - Request contains the final object generated in prior steps
  - endpoint must be configured to accept barebones TLS request and uses the root cert on the server end
  - serverside GET functionality defined later on in this document
- Request returns a certificate unique to the device in .pem format
  - Device stores this securely
- Device parses certificate for the expiration date and stores this value long term

#### Serverside GET Functionality

- Packet recieved from device
- Hash the {device id, current unix timestamp, CSR} against the device serial number(retrieved from AWS secrets manager) using HMAC-SHA-512
- Ensure the provided hash in the packet matches the calculated hash, throw out if they do not match
- Ensure the timestamp is recent by checking the current unix timestamp and comparing to the provided one
  - 2 minute buffer window is most likely sufficient, but can be lowered if the connectivity can handle it
  - Throw out if outside our time window
- Give the CA the CSR and get the output certificate
  - Certificate should be valid for 30 days from issue
- Update the date registered for the device in the registration table to the current timestamp if first time registration
- Update the certificate issue date in the registration table with the current timestamp
- Update cert_time_to_live in the registration table with our 30 day validity period
- Store the CSR in a bucket, can be reused later for valid renewals from a non-expired registration
- Return the certificate to the device

### Day to Day Device Operation With mTLS

#### Device Side

This uses the main device API that already exists

- mbedTLS can present the cert for any needed mTLS handshakes
- CURL requests should be appended with  --key `path-to-key` where this is the private key for the device, and --cert `path-to-cert` where this is the certificate obtained from registration
- Results returned from server work as normal
- Before sending the once-a-day trail update, the device performs a check of the current system time against the stored expiration date
  - If within 10 days of expiration or already expired, run the cert renewal procedure
  - If valid, use the api like normal
- If the device's certificate has problems on the serverside the server will return a 403 forbidden

#### Serverside

- mTLS is enabled on the AWS end for the api
- Verifies the certificate is properly set up, signed by our CA, and valid
- Device registration can be manually revoked from the devices page
  - Can also reenable devices
  - This cuts off a device from being able to send data to the server
  - Usable if a device has been stolen or just wants to be disabled without shutting off the physical device
- Expired or revoked certificate on a device attempting to use the main API will produce a 403 forbidden error which should be returned to the device

### Device Certificate Renewal

- Triggered when the device is within 10 days of expiration or already expired
- If cert is not expired but within the 10 days of expiration threshold, mTLS handshake to a special endpoint in the API
  - Server will use the CA to generate a new certificate valid for and additional 30 days
  - Registration table certificate issue timestamp updated to current time
  - Certificate returned to device, replacing old certificate
- If cert is expired, regenerate a new CSR and go through the full initial device registration process again as if it were a first time registration
  - This should only happen if a device is offline or out of internet range for over 10+ days prior to the expiration date and reconnects after the certificate expires
  - Device is functionally treated if it doesn't exist once the certificate expires
